# Масштабируемая архитектура для SFMShop

## Компоненты:
- **Load Balancer (Nginx):** распределяет запросы, входящие от пользователей, между несколькими API-серверами.
- **API-серверы (несколько экземпляров):** обрабатывают запросы юзеров, бизнес логику, запросы к бд.
- **Redis:** быстрый кэш для часто запрашиваемых данных (популярные категории, список товаров в частых запросах итд)
- **PostgreSQL с репликацией:** Основная БД принимает записи (POST / PUT), реплики для GET запросов.
- **Message Queue (RabbitMQ):** используется для асихнронных задач, позволяет API отвечать быстрее.
- **CDN для статики:** используется для отправки изображений, файлов итд. Хранит копии на разных серверах, отдаёт юзерам с ближайшего к ним.

## Масштабирование:
- **Горизонтальное масштабирование API:** API-сервисы масштабируются по RPS/CPU/latency. При достижении высокой нагрузки CPU или высокой latency добавляется новый экземпляр API. 
- **Вертикальное масштабирование БД:** основная постгрес БД получает больше CPU и RAM при росте изменяющих БД запросов. При росте GET запросов добавляются реплики и кэширование результатов.
- **Стратегия кэширования:** кэш в браузере юзера, CDN для статики, redis для API
- **Обработка пиковых нагрузок:** кэширование данных, автоматическое добавление API-серверов, очереди, ограничение запросов.

## Отказоустойчивость:
- **Health checks:** постоянные проверки состояния серверов. Если не отвечает - балансировщик перестаёт отправлять запросы к нему.
- **Автоматическое восстановление:** если API-серв падает, система автоматически запускает новый. Если БД падает, одна из реплик становится основной
- **Резервное копирование:** регулярное создание резервных копий Postgres и данных Redis.
- **Мониторинг и алертинг:** отслеживание RPS, latency, нагрузку на CPU, использование RAM. Уведомления при ненормальных показателях (с помощью специальных систем или хотя бы в телегу)

## Метрики:
- **Целевые метрики:** RPS: 5000+ запросов/сек, latency < 100 мс, availability > 99.9% (~9ч недоступности в году)
- **План масштабирования:** при увеличении нагрузки 1) добавляются новые API-сервера, 2) репликация БД, 3) использование Redis Cluster, увеличение кол-ва узлов, 4) добавляются новые очереди сообщений
